🧱 Building a Java 17 Development Pod

A detailed, learn‑as‑you‑go guide for creating a Java 17 dev environment in a container, wired up to VS Code—no Eclipse required.

This document walks you through:

  • Choosing the right base image (and why not Alpine for Java dev)
  • Writing a Dockerfile for OpenJDK 17 and dev tools
  • Adding your custom files into the image
  • Running the container as a dev pod
  • Connecting to it using VS Code Remote Containers
  • Optional: Maven/Gradle and devcontainer configuration

1. 🧩 Understanding the Components

Before building anything, it helps to know the main pieces involved:

Concepts

  • Docker Image – A template that defines your environment (OS, tools, configs). Learn more: What is a container/image?
  • Dockerfile – A script that describes how to build your image step by step. Learn more: Dockerfile reference
  • Container – A running instance of an image; think of it as a lightweight, isolated dev machine.
  • VS Code Dev Containers – VS Code connects into a container and treats it like a full development environment. Learn more: VS Code Dev Containers
  • OpenJDK 17 – The Java runtime and compiler you’ll use for development. Learn more: OpenJDK 17

2. 🧱 Choosing the Right Base Image

You asked whether using an Eclipse ISO or Alpine is better. For a Java development pod, especially one meant to be stable and VS Code–friendly, the best choice is:

Use an Ubuntu‑based Eclipse Temurin OpenJDK 17 image, not Alpine.

Why not Alpine for Java dev?

  • Alpine uses musl libc instead of glibc, which many Java tools expect.
  • Some JVM features and native dependencies can behave differently or break.
  • VS Code server and other tooling can be more fragile on Alpine.
  • You often end up debugging low‑level compatibility issues for minimal size savings.

If you’re building a runtime‑only image and are very size‑sensitive, Alpine can make sense. For a development environment, it usually isn’t worth the pain.

Recommended Base

  • Eclipse Temurin JDK 17 (Ubuntu-based) Learn more: Eclipse Temurin (Adoptium)

3. 🛠️ The Detailed Dockerfile (with Explanations)

Below is a teaching Dockerfile. Every section includes comments explaining why it exists. Save this as Dockerfile in a new directory. 

# Use Eclipse Temurin OpenJDK 17 (Ubuntu-based)
# This avoids Alpine's musl issues and ensures full Java compatibility.
FROM eclipse-temurin:17-jdk

# Install common development tools
# - git: version control
# - curl: downloading files
# - unzip: extracting archives
# - bash: VS Code server requires a shell
RUN apt-get update && apt-get install -y \
    git \
    curl \
    unzip \
    bash \
    && rm -rf /var/lib/apt/lists/*

# Copy your custom files into the image
# Replace "custom-config" with your folder name
# Example: configs, scripts, templates, etc.
COPY custom-config/ /opt/custom/

# Create a non-root user for VS Code
# VS Code Remote Containers prefers non-root for security and compatibility
RUN useradd -ms /bin/bash devuser

# Switch to the new user
USER devuser

# Set the working directory
WORKDIR /home/devuser

In the same directory as the Dockerfile, create a folder called custom-config and put any files you want baked into the image there.

4. 📦 Build the Image

From the directory containing your Dockerfile (and custom-config/), run:

docker build -t java-dev-env:latest .

This command:

  • Reads the Dockerfile
  • Builds an image with OpenJDK 17 and your tools
  • Tags it as java-dev-env:latest

Learn more about docker build: Docker build docs

5. ▶️ Run the Container (Your Dev Pod)

Now run a container from the image and mount your project into it. From your project directory: 

docker run -it --name java-dev \
  -v $(pwd):/workspace \
  -w /workspace \
  java-dev-env:latest

This does the following:

  • --name java-dev – Names the container java-dev so it’s easy to find.
  • -v $(pwd):/workspace – Mounts your current directory into /workspace inside the container.
  • -w /workspace – Sets the working directory inside the container.
  • -it – Runs interactively with a terminal.

Learn more about docker run: Docker run docs

6. 🧠 Connect Using VS Code (No Eclipse Needed)

Now we wire VS Code into this container so it becomes your full dev environment.

Install the VS Code extension

  • Extension: Remote - Containers
  • Marketplace link: Remote - Containers

Attach VS Code to the running container

  1. Open VS Code.
  2. Press F1 (or Ctrl+Shift+P / Cmd+Shift+P).
  3. Run: Remote-Containers: Attach to Running Container...
  4. Select java-dev from the list.

VS Code will:

  • Install its server inside the container.
  • Give you IntelliSense, debugging, terminals, etc.
  • Treat the container like a full dev machine.

More details: VS Code Dev Containers documentation

7. 📚 Optional: Add Maven or Gradle

If your Java projects use Maven or Gradle, you can bake them into the image.

Add Maven

Add this to your Dockerfile after the first RUN apt-get ... block (or merge it in):

RUN apt-get update && apt-get install -y maven \
    && rm -rf /var/lib/apt/lists/*

Learn more: Maven

Add Gradle

Similarly, for Gradle:

RUN apt-get update && apt-get install -y gradle \
    && rm -rf /var/lib/apt/lists/*

Learn more: Gradle

8. 📘 Optional: Add a Dev Container Config (Recommended)

You can make this setup even smoother by adding a .devcontainer/devcontainer.json file to your project. This tells VS Code exactly how to use your image.

Create a folder .devcontainer in your project, and inside it create devcontainer.json:

{
  "name": "Java Dev Pod",
  "image": "java-dev-env:latest",
  "remoteUser": "devuser",
  "workspaceFolder": "/workspace",
  "customizations": {
    "vscode": {
      "extensions": [
        "vscjava.vscode-java-pack"
      ]
    }
  }
}

This configuration:

  • Tells VS Code to use the java-dev-env:latest image.
  • Uses the devuser user we created in the Dockerfile.
  • Sets the workspace folder to /workspace.
  • Automatically installs the Java extension pack in VS Code.

Learn more: Dev Containers in VS Code

9. 🎯 Final Thoughts

You now have a clear, repeatable setup for a Java 17 development pod:

  • A Dockerfile that uses a solid, Java‑friendly base image.
  • OpenJDK 17 plus common dev tools installed.
  • Your custom files baked into the image under /opt/custom.
  • A container you can attach to from VS Code instead of using Eclipse.
  • Optional Maven/Gradle and devcontainer.json for a smoother workflow.

From here, you can extend this setup with debugging configs, Kubernetes manifests, or CI builds that use the same image—so your dev pod and your build environment stay in sync.
